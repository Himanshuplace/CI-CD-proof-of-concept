I
Chapter one · the map

The whole picture, before we zoom in.

Five distinct systems collaborate to take code from a developer's laptop to a running pod in production. Each one is a process you can debug independently — that separation is the entire point.

The diagram on the next page is the only diagram you really have to internalise. Everything else in this guide is a deeper look at one of its zones. Five colours, five concerns, and seven numbered hops connecting them. The numbers indicate the order in which things happen when a developer pushes code.

Notice three structural choices baked into this drawing. First, the CI pipeline never talks to Kubernetes — it pushes to a registry and pushes to a Git repo. Nothing else. Second, the gateway (HAProxy + KrakenD) lives outside the cluster, on a bare-metal VM. A Kubernetes outage doesn't take the gateway with it. Third, secrets travel from Vault to pods through a controller inside the cluster — the pod never reaches out to Vault directly.

The three bands, in plain English

Source & Build is everything before a deployable artifact exists. The developer pushes. CI runs. An image is built, scanned, signed, and pushed to a registry. At the end of this band, you have a single immutable thing — a container image — tagged with the exact commit it was built from. That image is what gets deployed; it's never rebuilt.

Orchestration is the contract between "what should be running" and "what is running." The GitOps repository is the source of truth for the former. ArgoCD is the agent that reconciles the latter to match. CI writes to the GitOps repo. ArgoCD reads from it. CI never reads the cluster; ArgoCD never reads CI. The two systems are decoupled, which is the whole point.

Runtime is where the pod actually serves traffic. Pods talk to Vault only indirectly, through the External Secrets controller. Pods receive traffic only via the gateway, which terminates TLS and enforces rate limits before anything touches business logic.

The pipeline never kubectl applys anything. That single rule shapes the rest of the architecture.

That rule — CI never touches the cluster directly — is what makes rollback a git revert instead of a frantic redeploy. It's what gives you an audit trail of every production change without any extra tooling. It's what lets ArgoCD detect and reverse drift if someone bypasses the pipeline. Almost every advantage of GitOps follows from that one constraint.

The flip side: ArgoCD is now a critical dependency. If ArgoCD itself is broken, you can't deploy. Production-grade installations run ArgoCD HA. Phase 1 of the migration plan (chapter IX) covers how to roll out ArgoCD safely.
II
Chapter two · the inputs

Git, branches, and the commits that drive everything.

Every pipeline run starts with a Git event. The structure of branches and commit messages is what makes the whole system legible — to humans and to automation.

A pipeline reads three things from Git: which branch you pushed, what commits you added, and what tag (if any) you created. The branch determines which deployment target the pipeline aims at. The commit messages drive semver versioning and changelog generation. Tags trigger production releases. These three signals carry an enormous amount of meaning — but only if they follow a discipline. That discipline is what this chapter codifies.

The model is trunk-based development: one protected main branch, short-lived feature branches that merge in and disappear, and semantic version tags that mark production releases. There is no long-running develop branch, no release branches, no GitFlow ceremony. For a team shipping multiple times per day, trunk-based is the only model that doesn't drown you in merge conflicts.

Branch types and what each promises

PrefixForLifetimeExample
feature/New functionality< 2 daysfeature/jwt-refresh
fix/Bug caught in development or staging< 1 dayfix/nats-reconnect
hotfix/Production incidentHourshotfix/payment-nil-ptr
chore/Dependency bumps, config, tooling< 1 daychore/bump-go-1-22
refactor/Restructure without behaviour change< 2 daysrefactor/extract-jwt
docs/Documentation onlyHoursdocs/runbook-update

The names aren't just labels. They're machine-readable signals. The CI pipeline checks branch names against a regex; feature/* branches deploy to a development environment, main deploys to staging, and only semver tags trigger production deploys. The branch prefix is also what tells GitLab what label to apply and what reviewers to auto-assign through CODEOWNERS.

Commit messages, and why they matter

Conventional Commits is the standard. Each commit starts with a type (feat, fix, chore, refactor, perf, test, docs, ci), an optional scope in parentheses, and a short subject line. The type drives the semver bump on the next release: feat bumps the minor version, fix bumps the patch, and BREAKING CHANGE: in the footer bumps the major. 

feat(payment): add NSDL mandate initiation flow
fix(nats): reconnect after JetStream stream timeout
chore(deps): bump grpc to v1.64.0
refactor(depository): extract CDSL/NSDL adapter interface
perf(redis): use pipeline for batch session operations
docs(adr): record ArgoCD migration decision
The reason for this discipline isn't aesthetics. It's that tools — commitlint, release-please, conventional-changelog — parse these strings. If commit messages are "fix stuff" and "more changes", none of those tools work. You write changelogs by hand, version bumps by hand, and the cost accumulates forever.

Where the rules are actually enforced

Three layers of enforcement, from cheapest to most expensive:

  1. Pre-commit hooks — local, instant

    Configured in .pre-commit-config.yaml. Runs gofmt, go vet, conventional commit format check, and Gitleaks secret detection before the commit even lands in your local Git history. Removes ~70% of trivial CI failures.

  2. CI pipeline — on push, ~10 seconds for the validate stage

    Same checks as pre-commit, plus branch-name regex, plus broader scope (the whole tree, not just changed files). This is the guard against developers who skipped pre-commit install.

  3. GitLab protected branch — at merge time

    The MR cannot be merged unless the pipeline is green, at least one approver has signed off (CODEOWNERS auto-assigned for sensitive paths), and all conversations are resolved. No direct pushes to main, even by maintainers.

III
Chapter three · the engine

Eight stages, in the order they're cheapest to run.

The pipeline is the part most engineers spend the most time staring at. Its design — what runs in parallel, what gates what, what fails the build — is what determines whether CI feels lightning-fast or like wading through molasses.

The single guiding principle of pipeline design is cheap rejections first. A typo in a Go file should fail in two seconds, not seven minutes. A pipeline that runs expensive tests before lint is a pipeline that wastes runner minutes on code that was never going to pass anyway. The eight stages below are arranged so that anything which can fail fast, does.

Click any stage to explore its jobs

Validate — the gauntlet of cheap checks

Four jobs run in parallel. None take more than a few seconds. If your code fails any of them, no expensive work has been wasted.

jobcommitlint
Validates every commit added by this MR against Conventional Commits. Catches: "fix stuff" subject lines, missing scopes, malformed messages. Without this, your changelog generation breaks silently.
jobbranch-name
A single grep regex against $CI_COMMIT_BRANCH. Catches: branches like my-test or tmp-fix. The format feature/short-description is enforced so downstream automation (auto-labelling, ticket linking) works.
jobformat-vet-lint
gofmt: formatting · go vet: suspicious constructs the compiler doesn't catch · golangci-lint: 20+ linters in one pass. The trio catches stylistic, semantic, and architectural smells respectively.
jobsecret-scan
Gitleaks scans the working tree for accidentally committed secrets — PEM headers, API tokens, anything matching the rules in .gitleaks.toml. The --redact flag keeps any found secret out of the CI log.

Test — proof the code does what it claims

One job. Two flags that matter enormously.

jobunit-test

go test -race -covermode=atomic ./...

-race instruments every memory read and write. If goroutine A reads a variable goroutine B is writing without synchronisation, the race detector prints an error and the test fails. This is exactly why every shared map in the service code uses sync.RWMutex — without it, the race detector would fail the build.

covermode=atomic uses atomic operations to count coverage. The simpler count mode is not thread-safe and produces wrong numbers in parallel tests.

Security — three tools, three threat models

Each tool catches a different class of problem. Running all three is the difference between "we have security scanning" and "we actually catch things."

jobgosec
Static analysis of your source code's AST. Catches: SQL injection patterns, hardcoded credentials, weak crypto (MD5, RC4), unsafe Go idioms. Reports as JSON — uploaded as an artefact so the security team can audit findings later.
jobgovulncheck
Cross-references your call graph against the Go vulnerability database. Catches: CVEs in dependencies where your code actually calls the affected function. Much lower false-positive rate than naive dependency scanning — if you import a library but don't use the vulnerable function, govulncheck doesn't flag it.
jobtrivy-fs
Scans go.mod, go.sum, and the Dockerfile for known vulnerable packages. Catches: what govulncheck doesn't — including vulnerabilities in base-image OS packages. --ignore-unfixed filters out CVEs with no available fix (failing a build on an unfixable CVE is noise).

Build — once, with cache, signed implicitly

A single Docker build produces an image tagged with the commit SHA. The image is never rebuilt for another environment — it's promoted by tag.

jobbuild-image

Multi-stage Docker build. The first stage uses golang:1.22-alpine with --mount=type=cache for the Go module and build caches. The second stage starts from gcr.io/distroless/static-debian12:nonroot — about 3MB. No shell, no package manager, non-root user.

Image tag is $CI_COMMIT_SHORT_SHA — never latest. latest can't be rolled back. abc123f is permanent and traceable.

OCI labels (org.opencontainers.image.revision, source) are baked into the image at build time. ArgoCD and your SBOM both reference them as provenance.

Package — supply chain security

Scan the built artefact. Generate a software bill of materials. Cryptographically sign the image. This is the layer that earns regulatory audit.

jobtrivy-image
Scans the BUILT image (not just source). Catches: vulnerabilities in base-image OS layers that trivy-fs couldn't see. With a distroless base image, this is usually empty — which is the point.
jobsbom-and-sign

syft walks the image and produces a JSON listing every library, version, and license. Stored as a 90-day artefact for compliance.

cosign sign --yes uses Sigstore's keyless flow: GitLab provides an OIDC token, Sigstore issues a short-lived certificate, the image is signed, and the signature lands in Sigstore's transparency log. No key to manage.

cosign attest produces a signed claim: "this image was built by this exact GitLab pipeline, from this exact commit, at this time." Auditors can verify this independently.

Deploy — a git push, not a kubectl apply

The pipeline doesn't deploy anything. It updates a YAML file in a separate GitOps repository. ArgoCD does the actual deploy.

jobdeploy-dev
Runs on every feature/* branch. Clones the GitOps repo, updates environments/dev/<service>/values.yaml with the new image tag, commits, and pushes. ArgoCD auto-syncs the dev environment within 10 seconds.
jobdeploy-staging
Runs only on merge to main. Updates environments/staging/. Same mechanism — but the GitOps commit message includes the MR number for traceability.
jobprepare-production
Runs only on semver tags (v1.2.3). Updates environments/production/ with the image tagged by the semver, not the SHA. ArgoCD requires manual sync for production — a human must click Sync, providing the audit trail.

Observe — verify the deploy actually worked

A GitOps push is not a deploy. Until /healthz returns 200 from a freshly-rolled pod, you don't actually know if the deploy succeeded.

jobsmoke-test-staging
Polls $STAGING_URL/healthz every 5 seconds for up to 120 seconds. If it never returns {"status":"ok"}, the pipeline fails. This is the gate that catches "ArgoCD never synced" or "new image crashes at startup" — failures invisible to the rest of the pipeline.
jobtrack-dora
Posts a deployment event to the DORA webhook only if the smoke test passed. The four DORA metrics — deploy frequency, lead time, change failure rate, mean time to restore — are calculated from this event stream.
jobnotify-deploy / notify-failure
Slack notifications. notify-deploy on success, notify-failure with when: on_failure guarantees the team hears about failed deploys even when other jobs are still running.

Agentic — AI review as a sidecar, never a gate

AI review is opinionated and useful. It's also fallible and depends on an external API. Treating it as advisory rather than blocking is what makes it deployable at all.

jobai-review

Runs only when the MR touches sensitive paths (auth, payment, CI config, charts, gitops, policies, go.mod). For everything else, the team's human reviewers handle review without AI assistance — which saves API budget on the 80% of MRs that are routine.

Sends the diff to Claude with a system prompt tuned for Go and the financial domain. The response is posted as an MR comment.

allow_failure: true means even if Claude API is down, the pipeline still passes. AI review never blocks a merge.

IV
Chapter four · the engine room

One Go binary, three protocols, two listening ports.

The service is the thing being deployed. Its shape — how protocols are organised, how state is shared, how requests flow — determines everything downstream.

The demo service in this repository deliberately speaks three protocols from one binary: REST over HTTP, gRPC over HTTP/2, and WebSocket over an HTTP upgrade. This is unusual — most production services pick one and stick to it. The reason all three are here is pedagogical: the architecture has to accommodate every common protocol your team is likely to use, and showing them together makes the trade-offs visible.

Why one binary instead of three

You could split each protocol into its own binary, each its own container, each its own deployment. Many teams do. The reason this service doesn't is operational weight: three containers means three sets of metrics, three rollouts, three ingress paths, three sets of resource limits to tune. For a five-person team shipping multiple times per day, that's three times the maintenance for no functional gain.

The shared in-memory store earns its keep here. An HTTP POST /api/v1/users immediately makes the new user visible to a subsequent gRPC ListUsers call — no Redis, no Postgres replication lag, no eventual-consistency window. If gRPC traffic outgrows the binary or the store gets too big for memory, split then. Not before.

A real production service replaces this in-memory store with a Postgres or Redis client. The interface is the same — the handlers don't know or care. That's the point of keeping the store as a separate package: it's a swap, not a rewrite.

The Dockerfile, briefly

The image is built in two stages. The first stage uses golang:1.22-alpine (about 350 MB) to compile a static binary. The second stage starts from gcr.io/distroless/static-debian12:nonroot — roughly 3 MB — and copies just the binary across. The final image has no shell, no apt, no curl, no ls. An attacker who achieves code execution inside the container has almost nothing to work with. 

# builder stage — discarded after the COPY below
FROM golang:1.22-alpine AS build
COPY . .
RUN CGO_ENABLED=0 go build \
    -trimpath -ldflags="-s -w" \
    -o /out/service ./cmd/server

# final image — what production runs
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /out/service /app/service
USER nonroot:nonroot
ENTRYPOINT ["/app/service"]

-trimpath removes local filesystem paths from the binary (no /home/runner/... baked in). -ldflags="-s -w" strips the symbol table and DWARF debug info — about 30% smaller binary. CGO_ENABLED=0 ensures a fully static build, which is required for the distroless base (no glibc).

V
Chapter five · the front door

HAProxy and KrakenD, both outside the cluster.

A two-tier gateway is a choice — most teams use one. Understanding why this architecture chooses two tiers, both outside Kubernetes, is the most important thing in this chapter.

The naive setup is to put a Kubernetes Ingress (nginx, Traefik) at the edge, terminate TLS there, and route directly to pods. It works. Many companies run their whole stack this way. The reason this architecture doesn't — and why HAProxy and KrakenD both live outside the cluster on bare metal — comes down to blast radius and operational independence.

Why two gateway tiers

The reason for two tiers rather than one is separation of concerns plus blast-radius isolation. HAProxy deals in TCP: connections, TLS handshakes, health probes, raw packets. It's small, fast, and stable — most installations run for years without a restart. KrakenD deals in HTTP semantics: route patterns, rate-limit windows, JWT validation, request/response shaping. It changes more often because the API surface changes more often. Keeping them separate means a KrakenD config error doesn't drop existing TCP connections — HAProxy keeps serving while KrakenD restarts.

Why outside Kubernetes

The other choice — putting the gateway inside the cluster as an Ingress — is simpler and well-supported. The reason this architecture goes the other way is that the gateway becomes useless when the cluster is down. If the entire K8s control plane has an outage (an actual upgrade incident, a network partition, a bad CNI rollout) and your gateway lives inside, you can't even serve a maintenance page. With the gateway on a separate VM, HAProxy can be reconfigured to route to a static "we'll be back soon" page or a backup cluster without touching Kubernetes at all.

The trade-off: the gateway VM is now state you have to manage outside K8s. That means separate monitoring, separate patching, separate runbook. If your team is already stretched thin, an in-cluster Ingress is the right call. The bare-metal gateway is a deliberate choice for teams that can absorb the extra operational surface in exchange for fault isolation.
VI
Chapter six · the contract

GitOps: how a git commit becomes a running pod.

The pipeline doesn't deploy anything. It updates a YAML file. ArgoCD does the rest. That separation is what makes the whole system trustworthy.

The GitOps deploy loop is the second most important diagram in this guide after the system overview. It shows how a single piece of state — the image tag in a YAML file — travels from the CI pipeline into the cluster, with the entire transit recorded in Git. Every box on the diagram is a process you can debug independently. Every arrow is something you can trace through a log.

Staging is automatic, production is manual

The ArgoCD Application object for staging has syncPolicy.automated: {prune: true, selfHeal: true}. That tells ArgoCD to auto-deploy any new commit, and to revert any drift (someone running kubectl apply directly) within the reconciliation window. Staging should always exactly match what's in Git, and never diverge.

Production deliberately omits the automated block. ArgoCD detects the difference between cluster and Git, but doesn't apply it until a human clicks Sync (or runs argocd app sync from the CLI). That human action is the audit trail. For a financial-domain deployment, having a recorded human approval per production deploy is often a compliance requirement.

Rollback is git revert. Audit trail is git log. Everything else follows from those two facts.

Tuning the reconciliation loop

ArgoCD's default timeout.reconciliation is 180 seconds. For a team shipping multiple times per day, that's a noticeable wait. Drop it to 10 seconds: 

# argocd-cm ConfigMap
data:
  timeout.reconciliation: 10s
  timeout.hard.reconciliation: 0s  # disable forced full re-sync

At 10 seconds, the average time from a GitOps push to ArgoCD picking it up is about 5 seconds. Combined with a 30-second rolling deploy, you're looking at under a minute from CI completion to traffic on the new pod.

VII
Chapter seven · the keys

Vault, External Secrets, and the pod that never knows.

Secrets are different from configuration. They rotate. They need audit. They have blast radius. The architecture for them is its own concern, distinct from how the pod gets its config.

A pod needs a database password. The naive way to give it one is to put it in a Kubernetes Secret manifest and commit it (encrypted or not) to Git. That works. It's also bad: rotation requires a commit, the secret value appears in Helm rendering, and there's no audit trail of who read it when. The path this architecture takes — Vault, via the External Secrets controller, into a Kubernetes Secret, finally read by the pod as an env var — looks longer but is materially better at every step.

When NOT to use this

Not every secret needs Vault. The Slack webhook URL that posts a CI notification isn't worth the operational overhead of Vault. The pull token for a public container registry isn't either. For those, a GitLab CI variable is perfectly fine. The decision rule is:

PropertyGitLab CI variableVault + ESO
Setup time5 minutesHours to days
Audit log (who read what, when)NonePer-access in Vault
RotationManualAutomatic, possibly dynamic
Dynamic credentials (DB user per pod)Not possibleNative
Use forWebhook URLs, public API keys, build tokensDB passwords, signing keys, PII access
For a regulated domain like financial services, "audit log of secret access" isn't a nice-to-have — it's what makes you survive an external audit. Vault is worth the setup cost the moment you have a regulator-visible secret.
VIII
Chapter eight · the people

The rules that aren't in code.

Tools enforce mechanics. Norms enforce judgement. Both are necessary. Both are written down.

A pipeline can validate commit messages. It can run tests. It cannot tell you when to revert versus fix forward, who should review which MR, or how to handle a production incident at 2 AM. Those are team norms, and they have to be written down or they don't exist. The contents of CONTRIBUTING.md in this repo are the team's operating manual.

The three non-negotiables

First rule

Main is always deployable

Every commit on main is a potential production release. If you wouldn't deploy it right now, don't merge it.

Second rule

You broke main, you fix main

Within 30 minutes. Fix forward or revert. If you can't, ping the team and someone else reverts.

Third rule

Risky changes ship behind a flag

Payment logic, auth flows, depository adapters. Feature flag is OFF on merge. Decouple deploy from release.

The review model

One approver, four working hours. CODEOWNERS auto-assigns reviewers on sensitive paths (auth, payment, CI config, charts, gitops, policies, supply chain). Everything else can be reviewed by any teammate. There's no requirement for two approvals on most paths — for a five-person team, that's bureaucratic. Trust the team and the pipeline.

What the author owes the reviewer: a self-reviewed diff with no debug logs, a passing pipeline, and an MR description that fills the template — risk level set, rollback path documented for medium and high risk changes.

What the reviewer owes the author: feedback within four hours, focused on correctness and rollback safety rather than personal style preferences. If you would have written it differently but the diff is correct, approve it.

Hotfix procedure, in shell

# production incident — tested-and-shipped within 1 hour

$ git checkout main && git pull
$ git checkout -b hotfix/payment-nil-ptr

# minimal fix + a test that fails without it
# do NOT refactor anything nearby; do NOT cleanup

$ git push origin hotfix/payment-nil-ptr
# fast-track MR — any one approver, Slack-ping
# merge to main

$ git tag v1.4.1
$ git push origin v1.4.1
# pipeline updates production GitOps repo
# argocd app sync production-payment-svc
# verify

# postmortem within 24h — docs/incidents/YYYY-MM-DD-...
Hotfixes skip the normal "two days in staging" sanity check. That's the explicit trade-off — speed in exchange for safety. To balance the books, hotfixes require a postmortem within 24 hours. The postmortem is what extracts learning from the incident, so it doesn't happen again.
IX
Chapter nine · the path

Migrating from helm push to all of this.

If you're starting from a working pipeline that just does helm upgrade from CI, this is the four-phase route to everything described above.

The single biggest mistake teams make when adopting GitOps is doing it all at once. ArgoCD plus Vault plus External Secrets plus Kyverno is four new tools for an operations team that's "mostly new" to any of them. Trying to absorb all four in a quarter is how you end up with a half-working migration and a team that's lost trust in the new system. The phased plan below shipped value at each step, and at any point you can stop and still be better off than where you started.

  • Phase 0 · Week 1

    Right-size team norms

    Simplify CODEOWNERS for a small team. Add pre-commit hooks. Write CONTRIBUTING.md. Record the migration plan as an ADR.

    Done
  • Phase 1 · Weeks 2–4

    ArgoCD adoption with dual-deploy safety net

    Service by service. CI does BOTH helm upgrade (existing path) AND git push to gitops-config (new path). ArgoCD is in audit-only mode for one week — it logs what it would do but doesn't act. After cluster state matches ArgoCD's "would-deploy" state for a week straight, flip to auto-sync. After auto-sync is stable for another week, drop the helm-push step.

  • Phase 2 · Weeks 5–6

    Feature flags and auto-rollback

    Stand up Flagsmith (self-hosted, one container plus Postgres). Wrap the next two features in flags. Add auto-rollback: smoke-test-staging failure triggers argocd app rollback automatically. No human in the rollback loop.

  • Phase 3 · Months 2–3

    Vault and External Secrets

    Stand up Vault in-cluster (HA, three replicas). Install External Secrets Operator. Migrate ONE non-critical secret first (a build token, not a DB password). Verify the round-trip. Then migrate the next. Don't bulk-migrate.

  • Phase 4 · Month 3+

    Kyverno policy enforcement

    Install Kyverno. Apply the policies in audit mode first for two weeks — they log violations but don't block. Once the team understands what's flagged (often: someone is running an unsigned image, or a Pod missing resource limits), switch to enforce mode. Suddenly nobody can deploy non-compliant pods.

You can stop at the end of any phase and still be better off than where you started.

That property — each phase ships value independently — is what makes a four-phase plan viable. If the team needs to pause and focus on shipping features for a month, you can. The half-migrated state is functional. The full state is just incrementally better.
