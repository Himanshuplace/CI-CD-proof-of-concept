00

The big picture

Five distinct systems collaborate to take code from commit to running in production. Each box below is a process you can debug independently — that separation is the whole point of the architecture.

Data flow
Code travels left to right along the top row, then drops into the GitOps repo, then ArgoCD pulls and deploys. CI never touches Kubernetes directly — only git push + image push.
Gateway tier
HAProxy and KrakenD live outside Kubernetes on a bare-metal VM. A K8s outage doesn't take down the gateway. The gateway can route to multiple clusters if you grow that far.
Secrets flow
The pod never knows Vault exists. External Secrets Operator syncs Vault → K8s Secret. The pod reads it as an env var like any other. Replace Vault later and nothing in the pod changes.
Policy enforcement
Kyverno runs as a Kubernetes admission webhook. It rejects pods that aren't signed, run as root, or pull from an unauthorised registry. The policy is code, version-controlled like everything else.

Git workflow

Trunk-based development. One protected main branch. Short-lived feature branches. No develop branch, no release branches. Designed for multi-deploy-per-day cadence.

Commit message format

Conventional Commits. Enforced by commitlint in CI and as a pre-commit hook. The type drives semver: feat bumps minor, fix bumps patch, BREAKING CHANGE bumps major. 

feat(nsdl): add eDIS mandate initiation flow
fix(nats): reconnect after JetStream stream timeout
chore(deps): bump grpc to v1.64.0
refactor(depository): extract CDSL/NSDL adapter interface
perf(redis): use pipeline for batch session operations
docs(adr): record ArgoCD migration decision
Enforced in CI
  • Branch name regex (job: branch-name)
  • Commit message format (job: commitlint)
  • Pipeline green required for merge
  • 1 approver required (CODEOWNERS for sensitive paths)
Enforced locally
  • gofmt, go vet via pre-commit hooks
  • Conventional commit format at commit-msg stage
  • Secret detection (gitleaks) before push
  • Large file blocker (>500KB)

The pipeline

Eight stages. Cheap rejections first, expensive checks later. Click a stage to expand the jobs and why each one exists. The full definition lives in ci-templates/templates/go-service.yml.

→ → → → → → →

01 — validate

Cheap checks that fail in seconds. Catches 90% of trivial breakages before any expensive work runs.

commitlint
Validates every commit in the MR follows Conventional Commits. Without this, automated changelog and semver bumping break.
branch-name
Regex check that the branch matches the team's naming convention. Catches "my-test-branch" before it gets reviewed.
format-vet-lint
gofmt -l ensures formatting · go vet catches bugs (wrong printf verbs, mutex copies) · golangci-lint runs 20+ linters in one pass.
secret-scan
Gitleaks scans the working tree for committed secrets (PEM keys, API tokens). --redact keeps the secret value out of the CI log.

02 — test

Unit tests with the race detector. Coverage gate prevents "we'll add tests later" from becoming permanent.

unit-test
go test -race -covermode=atomic. -race instruments memory access — catches concurrent bugs that would silently corrupt prod data. covermode=atomic is the correct mode for tests using t.Parallel().
coverage gate
awk extracts the total percentage and fails if below the threshold (75% default). Tunable per service via COVERAGE_THRESHOLD.

03 — security

Three different threat models. Each tool catches what the others miss.

gosec
Static analysis of YOUR source. Catches: SQL injection patterns, hardcoded credentials, weak crypto, unsafe ops.
govulncheck
Cross-references your call graph against the Go vuln DB. Only flags CVEs in code you actually CALL — much lower false-positive rate than naive dep scanning.
trivy-fs
Scans go.mod/go.sum + Dockerfile for vulnerable packages. --ignore-unfixed filters CVEs you can't act on.

04 — build

Multi-stage Docker build. The final image is distroless: ~3MB, no shell, non-root.

build-image
Uses needs: on test+security jobs (not blanket stage gate) so build starts the moment its dependencies are done. Tags image with commit SHA — never latest, because latest can't be rolled back.
FROM golang:1.22-alpine AS build
RUN CGO_ENABLED=0 go build -trimpath -ldflags="-s -w" ...

FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /out/service /app/service
USER nonroot:nonroot

05 — package

Supply chain security. Scan, sign, attest. Required for financial-domain compliance.

trivy-image
Scans the BUILT image (not just source). Catches CVEs in base image layers that trivy-fs missed.
sbom-and-sign
syft generates SBOM (list of every library in the image) · cosign sign --yes keyless signing via Sigstore OIDC · cosign attest signs a claim: "this image was built by this pipeline, from this commit."
Why this matters
When a CVE drops (Log4Shell, etc.), security asks "which services use library X?" — query the SBOMs. Answer in seconds, not hours.

06 — deploy

CI never touches Kubernetes. It only updates a YAML file in the GitOps repo. ArgoCD does the actual deploy.

deploy-dev
Runs on every feature/* branch. Updates the environments/dev/ values file.
deploy-staging
Runs on merge to main. Updates environments/staging/. ArgoCD auto-syncs within 10 seconds.
prepare-production
Runs on semver tag (v1.2.3). Updates environments/production/ but ArgoCD requires manual sync — human gate.

07 — observe

Verify the deploy actually worked. Tell the team. Record metrics.

smoke-test-staging
Polls /healthz for up to 120 seconds. If it never returns 200, fail the pipeline. This is the gate that catches "ArgoCD never synced" or "the new image crashes on startup."
track-dora
POSTs a deployment event to the DORA webhook. Only fires if smoke test passed — DORA shouldn't count failed deploys.
notify-deploy / notify-failure
Slack notifications. Success: green with commit link. Failure: red with pipeline link. when: on_failure ensures the alert fires even when other jobs failed.

08 — agentic

AI code review. Advisory only — humans decide. Restricted to sensitive paths to save API budget.

ai-review
Posts a Claude-generated review comment on the MR. Triggers only when MR touches auth, payment, go.mod, CI config, charts, or policies. allow_failure: true — never blocks a merge.
Why advisory only
In a financial system, AI cannot be in the critical decision path. If Claude API is down, you still need to merge. If Claude flags something, humans verify.
03

The service

One binary, three protocols, two listening ports. Shared in-memory store (sync.RWMutex-protected) backs all three protocols.

HTTP / REST
Standard library only (Go 1.22 native path params: /users/{id}). No Gin, no Echo. Less deps = smaller image, faster build.
service/internal/handler/http.go
WebSocket
gorilla/websocket upgrades HTTP → WS on /ws/live. Pushes a tick message every second. Read deadline + context cancellation handle disconnects.
service/internal/handler/ws.go
gRPC
google.golang.org/grpc with a custom JSON codec. Skips protoc for the demo. Real .proto file is committed — run make proto to generate stubs.
service/internal/handler/grpc.go

Why one binary instead of three?

Three protocols, one process, one container, one deployment, one set of metrics. Easier to manage. The shared in-memory store means an HTTP create is immediately visible to a gRPC list — no Redis, no DB sync needed. If gRPC traffic outgrows the binary, split it then. Not before.
04

Gateway layer

Two-tier gateway, both outside Kubernetes. HAProxy terminates the edge connection. KrakenD handles API logic. The Go service stays focused on business logic.

HAProxy — edge tier
  • TCP/TLS termination — handshakes complete here, backends speak plain HTTP
  • Health checks — every 3s, KrakenD is marked DOWN after 3 failures
  • WebSocket ACL — long-lived WS connections use timeout tunnel 3600s, bypass the 60s default
  • Stats page on :9000 — real-time request rate, error rate per backend
gateway/haproxy/haproxy.cfg
KrakenD — API tier
  • Rate limiting — per-endpoint, per-IP (100 rps total, 20 rps per IP)
  • CORS — single place to manage cross-origin policy for all services
  • Route matching — /api/v1/users/{id} → backend service
  • Aggregation — combine multiple backend calls into one client response
gateway/krakend/krakend.json
Why outside Kubernetes?
A Kubernetes control-plane outage takes down the cluster, but not the gateway. HAProxy on the host can show a maintenance page or route to a backup cluster. If you trust K8s availability completely, in-cluster Ingress (NGINX, Traefik) is simpler — but for financial domain with strict uptime SLAs, the extra independence of an external gateway is worth the operational cost.
05

GitOps deploy

The CI pipeline never runs kubectl apply. It only updates a YAML file in a separate Git repo. ArgoCD watches that repo and reconciles the cluster to match. This is the contract that makes rollback trivial (git revert) and audit logs free (git log).

Staging
syncPolicy.automated: { prune: true, selfHeal: true }
ArgoCD auto-deploys every commit to staging. Drift is reverted automatically.
Production
No automated: block. Deploy requires argocd app sync from a human. Audit trail auto-recorded.
Rollback
In GitOps repo: git revert the commit that updated the image tag. ArgoCD sees the revert within 10s, reverts the cluster.

07

Team norms

The rules that aren't in code, written down so they're enforceable. Right-sized for a 5-person team shipping multiple deploys per day. Full text in CONTRIBUTING.md.

The three non-negotiables

01
Main is always deployable
Every commit on main is a potential production release. If you wouldn't deploy it right now, don't merge it.
02
You break main, you fix main
Within 30 minutes. Fix forward or revert. If neither is possible, ping the team and someone else reverts your change.
03
Risky changes ship behind a flag
Payment logic, depository adapters, auth flows. Flag is OFF on merge. Decoupled deploy and release.

Review SLA

4 working hours
Maximum time an MR sits without a review. If you can't review in that window, reassign.

Hotfix procedure

Production incident. Tested and shipped within 1 hour.

# 1. branch from current main
git checkout main && git pull
git checkout -b hotfix/payment-nil-ptr

# 2. minimal fix + test that fails without it
#    do NOT refactor, do NOT clean up nearby code

git push origin hotfix/payment-nil-ptr

# 3. fast-track MR — any 1 approver (Slack ping is fine)
# 4. merge to main

# 5. tag immediately
git tag v1.4.1
git push origin v1.4.1

# 6. argocd app sync production-payment-svc
# 7. write postmortem within 24h (docs/incidents/YYYY-MM-DD-...)

CODEOWNERS — who reviews what

Right-sized: explicit ownership only for genuinely sensitive paths. Everything else can be reviewed by any teammate.

08

Migration plan

From "helm push from CI" to full GitOps + Vault + Kyverno, in four phases. Strangler-fig pattern: new system runs alongside old, takes over piece by piece. Each phase ships value independently. Full details in docs/adr/0001-cicd-migration-plan.md.

Phase 0
Week 1
Team norms + lighter review process
Right-size CODEOWNERS for 5 people. Add CONTRIBUTING.md. Add pre-commit hooks. Create first ADR. Outcome: trivial CI failures drop ~70%; team has documented norms.
✓ DONE
Phase 1
Week 2-4
ArgoCD migration with dual-deploy safety net
Service by service. CI does BOTH helm upgrade (existing) AND git push to gitops-config (new). ArgoCD in audit-only mode for 1 week. Verify match. Switch to auto-sync. Outcome: rollback = git revert, audit trail automatic.
Phase 2
Week 5-6
Feature flags + auto-rollback
Deploy Flagsmith self-hosted. Wrap next 2 features in flags. Smoke-test failure triggers argocd app rollback automatically. Outcome: deploy and release are decoupled; no human in the rollback loop.
Phase 3
Month 2-3
Vault + External Secrets
Stand up Vault in-cluster. Migrate ONE non-critical secret first (build token). Verify. Migrate the next. Outcome: audit log per secret access; rotation is automatic; pods unchanged.
Phase 4
Month 3+
Kyverno policy enforcement
Start in audit mode (log violations, don't block) for 2 weeks. Once team understands what's flagged, switch to enforce. Outcome: unsigned images, root containers, untrusted registries blocked at admission.
